# 案件管理システム

社内PM・営業担当者（5名程度）向けの案件管理Webアプリケーション。  
エクセル管理からの脱却を目的に、案件横断の進捗管理と情報の一元管理を実現する。

---

## 1. セットアップ手順（ローカルで動かす方法）

### 方法A: ローカル環境（Docker不要）

**前提条件:** Node.js 20 以上、npm

```bash
git clone <repository-url>
cd project-management-dashboard

# 依存パッケージのインストール
npm install

# 環境変数の設定
cp .env.example .env

# データベースのセットアップ（SQLite）
npx prisma migrate dev

# 開発サーバーの起動
npm run dev
```

### 方法B: Docker（Node.js不要）

**前提条件:** Docker / Docker Compose

```bash
git clone <repository-url>
cd project-management-dashboard

# ビルド＆起動（初回はビルドに1〜2分）
docker compose up --build
```

停止: `docker compose down`  
データも消したい場合: `docker compose down -v`

### 起動後の共通手順

1. ブラウザで http://localhost:3000 にアクセス
2. ログイン画面の **「デモデータ投入」** ボタンを押す
3. デモアカウントでログイン

| ロール | メールアドレス | パスワード |
|--------|-------------|-----------|
| 管理者 | admin@example.com | password123 |
| 担当者 | tanaka@example.com | password123 |

## 2. 技術スタックを選んだ理由

### フレームワーク: Next.js（App Router）
- **課題**: フロントエンドとバックエンドをどう構成するか
- **選択肢**: React + FastAPI（フロント/バック分離） or Next.js（フルスタック一体）
- **理由**: 今のところAPIを外部（モバイルアプリや他システム）から利用する想定がなく、社内ブラウザからの利用に限られる。汎用APIが不要な段階でフロント/バックを分離すると、API設計・CORS設定・デプロイ先の管理など不要な運用コストが増える
- **効果**: TypeScriptの型をDB定義からUI表示まで一貫して使えるため、データの受け渡しで型ズレが起きにくい体感はあり、AI使用する上でもシンプルかつ型定義がされるので、そこの問題は特に起きなかった印象

### ORM / DB: Prisma + SQLite
- **課題**: DB操作の型安全と環境構築の手軽さの両立
- **選択肢**: Prisma vs TypeORM / SQLite vs PostgreSQL
- **理由**: TypeORMより直感的に設計書のような形で記載が可能。Prismaはスキーマ定義からTypeScriptの型を自動生成でき、DBカラム追加時にコンパイルエラーで漏れに気付ける。マイグレーション機能も内蔵でスキーマ変更の管理が楽(これはTypeORMもいっしょなのか..?)。SQLiteはファイルベースで外部サーバー不要
- **効果**: 環境構築コストゼロで始められ、将来PostgreSQL等に移行する場合もPrismaの接続先変更だけで対応可能の想定なので、将来的にもいいかもしれない（ただ今回はローカルのみの動作なので、mysqlとpostgreSQL等より簡単だったことが大きい）

### 認証: NextAuth.js v5
- **課題**: 変更履歴に「誰が変更したか」を記録するには認証が必須
- **選択肢**: NextAuth.js vs 自前実装 vs Firebase Auth
- **理由**: 変更履歴に「誰が」を記録するには、変更操作時にログインユーザーのIDを特定する必要があったため。認証方式がプラグイン形式になっている。認証方式を変更する際はProviderの設定やログイン周りを修正するのみで、Action・Service・コンポーネント等のアプリ側コードは変更不要
- **効果**: 単に現在のログインユーザー情報が欲しかっただけなので大きな恩恵は得られていないが、認証方式の変更がアプリ全体に影響せずせず、認証周りのファイルに閉じた改修で済むという想定（不要な考慮な気もしている）

### データ取得: TanStack Query v5
- **課題**: フィルタ切替やデータ更新のたびに、取得中・エラー・再取得といった状態管理が必要になる
- **選択肢**: TanStack Query vs useState + useEffect（React標準のみ）
- **理由**: React標準のみで書く場合、データ取得のたびに`useState`で取得結果・ローディング・エラーを個別に管理し、`useEffect`で取得タイミングを制御するコードを毎回書く必要がある。TanStack Queryは「このパラメータでこのデータが欲しい」と宣言するだけで、取得・キャッシュ・再取得・ローディング表示を自動で管理してくれるといったことをするらしいが、あまり理解はできなかった。が、対象のデータが更新されたときキャッシュが古くなったと認識し、裏で自動的にAPIで再取得するという機構らしいので、useStateたちを使用するより状態の管理が楽そうと判断
- **効果**: あまり感じられていない。コードが的にはシンプルになったのかといったところ

### UI: Tailwind CSS + shadcn/ui
- **課題**: デザインの統一感を保ちたい
- **選択肢**: Tailwind + shadcn/ui vs MUI vs Chakra UI
- **理由**: shadcn/uiはソースコードごとコピーして使う形式で、プロジェクトに合わせて自由にカスタマイズ可能。MUIやChakra UIより使いやすそうだったのと少し慣れてたため
- **効果**: 統一感が保てた（スマホへのスタイル対応も考慮必要か）

### 設計: レイヤードアーキテクチャ
- **課題**: 将来のバックエンド分離や技術移行に備えた疎結合な構造が必要
- **選択肢**: レイヤードアーキテクチャ vs フラット構成（ページ内に全処理）
- **理由**: 処理を以下の3層に分離し、各層が独立して差し替え可能な構造にした
  - **Repository層**（`src/repositories/`）: DBへの読み書きだけを担当。例: `project-repository.ts` の `findProjects()` や `updateProjectData()` がPrisma経由でSQLiteを操作
  - **Service層**（`src/services/`）: ビジネスロジックを担当。例: `project-service.ts` の `updateProject()` が変更差分の計算や変更履歴の記録を行い、Repository層を呼び出す
  - **Action層**（`src/actions/`）: ユーザーからの入力検証と認証を担当。例: `project-actions.ts` の `updateProjectAction()` がログイン確認→Zodバリデーション→Service層呼び出しの順で処理
- **効果**: 例えばDBをSQLiteからPostgreSQLに変える場合はRepository層だけ、バックエンドをFastAPIに分離する場合はAction層をAPI呼び出しに差し替えるだけで済む。疎結合による恩恵はあるかと。

## 3. 必須要件以外に実装した機能と、追加した理由

### フェーズのマスター管理
課題では固定フェーズだったが、会議にて「自由入力ではフェーズ名がバラつく」との指摘。マスターテーブルで統一管理し、案件ごとに選択・順序変更可能とした。統一性と柔軟性の両立。

### 認証機能（NextAuth.js）
課題では「モックUIで擬似切替」とされていたが、変更履歴の「誰が」を記録するには認証が必須。プロバイダーパターンにより、将来のOAuth/SSO移行も設定変更のみで対応可能。

### 粗利の自動計算
受注金額・外注費・経費を管理する以上、粗利の自動算出は自然な拡張。DB保存せずアプリ層で算出し、データの一貫性を維持。

### 期限間近の警告表示（黄色）
必須は「期限切れの赤色」のみだが、7日前から黄色で警告することで予防的アクションを促す。

## 4. 実装しなかったが「あると良い」と思った機能と、その理由

### 案件検索（必須要件）
テキスト検索は必須要件だったが、時間の都合で未実装。フィルタ（ステータス・担当者）とソート（納期）は実装済み。

### スマホでのクイック確認
商談先や移動中にステータスをサッと確認できるスマホ最適化。Tailwindのレスポンシブで基本閲覧は可能だが、テーブルのカード表示切替等のスマホ固有最適化は時間制約で見送り。

### 週・月単位の差分可視化
社内MTGで「先週から何が変わったか」を一目で把握したい場面を想定。ActivityLogのフィールドレベル記録は実装済みなので、UIの追加のみで対応可能なデータ基盤は整えた。

### 通知・リマインダー
期限超過や期限間近の案件をメールやSlackで通知する機能。毎日画面を開かなくても対応漏れに気づける。商談が立て込んでいるときほど確認が後回しになりやすいため、プッシュ型の通知があると効果的。

```

├── src/
│   ├── app/              # Next.js App Router（ページ・API Routes）

│   ├── repositories/     # データアクセス層（Prisma）

│   └── types/            # 型定義
└── package.json
```
